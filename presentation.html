Problem Definition

What Are We Solving?

Daily Allocation Decision

Each trading day, allocate capital between two sleeves:

  • Market sleeve — S&P 500 + Dow proxy
  • IPO sleeve — market-cap weighted basket of recent IPOs (180-day post-IPO window)
Later expanded to per-sector IPO sleeves

Hard Constraints

  • Long-only (no short positions)
  • Fully invested (weights sum to 1.0)
  • No leverage

Why Is This Hard?

  • IPOs have high upside but unstable tails
  • Regime shifts: IPO return dynamics change across market cycles
  • Standard mean-variance fails when return distributions are non-stationary
  • Data quality issues: survivorship bias, stale sector metadata, delisted SPACs

Success Metrics

  • Sharpe ratio vs market-only baseline
  • Max drawdown control
  • Turnover and transaction cost awareness
  • Out-of-sample (walk-forward) performance
The question is not just "maximize return" — it is "get strong risk-adjusted return without blowing up concentration, tails, or instability."
Literature Review → Design Decisions

How the Literature Shaped Our Approach

Online Portfolio Optimization

Follow-the-winner, EG, CORN, and related algorithms. Key insight: sequential decisions under non-stationarity require adaptive objectives, not static ones.

IPO Anomaly & Risk Literature

Post-IPO underperformance (Ritter 1991), lock-up expiry effects, hot/cold IPO market cycles. Confirmed: IPO returns are regime-dependent and tail-heavy.

Risk-Aware Objectives

CVaR as a coherent risk measure (Rockafellar & Uryasev 2000), drawdown-penalized objectives. These directly informed our loss function design.

Sequence Models for Allocation

LSTM/GRU for time-series forecasting, attention mechanisms for regime detection. We evaluated GRU, LSTM, and Transformer variants.

The literature pushed us toward objective engineering, not just architecture chasing.

Prior work highlighted tail risk and objective mismatch. We translated that directly into explicit loss terms — this became the single biggest technical lever in the project.

Core Technical Contribution

Updated Multi-Term Loss Function

Old Objective

L_old = −E[r] + λ·Var(r)

Simple return minus variance. Ignored tails, concentration, and path stability.

New Objective

L = −E[r]
+ λ_cvar · CVaR_α(r)
+ λ_vol · σ(r)
+ λ_div · HHI(w)
+ λ_to · ‖Δw‖₁
+ λ_bm · max(0, r_bm − r)
+ λ_lg · (−log growth)

Tail Risk — CVaRα

Expected loss in the worst α% of days. Prevents the model from ignoring catastrophic scenarios that mean-variance misses.

Volatility Control — σ(r)

Penalizes daily return variance directly. IPO tails can spike volatility even when mean looks fine.

Diversification — HHI(w)

Herfindahl index on weights. Prevents collapse into a single sleeve (concentration risk).

Turnover — ‖Δw‖₁

Penalizes large day-to-day weight changes. Reduces transaction costs and produces smoother allocation paths.

Benchmark & Log-Growth

Downside relative to market; compound growth alignment. Optional terms for regime-specific calibration.

Objective design mattered as much as — or more than — model architecture in this project.
Training Engineering

Hyperparameters & Peripheral Training Changes

What We Varied

  • Learning rate schedules (cosine, linear warm-up, Noam)
  • Batch size: 32, 64, 128
  • Penalty weights: λ_cvar, λ_vol, λ_div, λ_to
  • Context window length (w = 63, 126, 252 trading days)
  • Gradient clipping threshold
  • Dropout (0.1, 0.2)
  • Weight decay (0, 1e-4)
  • Hidden dim (64, 128, 256)

Key Finding

"Small" settings moved results as much as big architecture swaps.

Batch size and λ calibration often had larger effect on Sharpe than switching GRU→LSTM.

WRDS ablation grid — each cell is one hyperparameter variant vs baseline

Stability Improvements

  • Gradient clipping: prevented loss spikes in early training
  • Cosine LR + warm-up: smoother convergence
  • Early stopping on validation Sharpe (not just loss)
Regularization is necessary, but too much kills useful signal — the right balance required empirical ablation, not theory alone.
Evaluation Design

Context Window · Walk-Forward · Training Up to the Day

Context Window Experiments

  • w = 63 (3 months): fast adaptation, higher noise
  • w = 126 (6 months): practical balance
  • w = 252 (1 year): smoother but slower to react

Window tuning gave mostly incremental gains unless coupled with objective changes.

Walk-Forward Setup

  • Expand training window forward in time
  • Evaluate on strictly future data only
  • No lookahead bias — each split uses only history available at that date
  • Reduces false confidence from a single static train/val split

"Training Up to the Day" (Online Framing)

  • Fit on all available history up to each evaluation point
  • Compare update vs no-update behavior
  • Measure benefit of fresh retraining vs frozen weights
Context window comparison — rolling Sharpe across w=63/126/252
Online cumulative returns — update vs no-update path
Walk-forward evaluation and temporal splitting made our claims more defensible and more honest than any single-split validation.
Model Architecture · LSTM

LSTM: What We Expected vs What Happened

LSTM Architecture

Input: rolling window features [T × F]
↓
LSTM encoder (gates: input, forget, cell, output)
↓
Hidden state h_T
↓
Linear → Softmax
↓
Allocation weights w ∈ Δ²

Cell state enables long-range memory — theoretically useful for multi-year IPO cycles.

What We Expected

  • Better long-term dependency capture vs GRU
  • Improved handling of multi-year IPO cycles
  • Reduced forgetting of historical regime context
GRU vs LSTM — training and validation loss curves

What Actually Happened

  • LSTM: Sharpe 3.70, MaxDD −7.19%, TotalRet 95.1%
  • GRU: Sharpe 3.57, MaxDD −7.27%, TotalRet 79.1%
  • LSTM showed marginal gains — not a universal win
  • Longer training time, higher memory, similar stability
  • Gains were conditional on objective calibration — not architecture alone

We include LSTM to show rigorous comparison, not to over-claim. The GRU remains our practical baseline.

LSTM gave small, conditional improvements. Architecture swap alone did not explain performance — objective calibration remained dominant.
Model Architecture · Transformer

Transformer: Why We Tried It, What We Found

Transformer Architecture

Input: rolling window features [T × F]
↓
Positional encoding + Multi-head Self-Attention
↓
Feed-forward layers × N
↓
CLS token / mean pooling
↓
Linear → Softmax → w ∈ Δ²

Motivation

  • Attention could directly focus on regime-change dates
  • Parallel computation over the full window
  • State-of-the-art on many time-series benchmarks
Transformer ablation variants — loss curves across settings

Honest Result

  • Did not consistently outperform GRU in risk-adjusted terms
  • Significantly higher compute and memory cost
  • Prone to overfitting on our relatively small dataset
  • Gains in training loss did not translate to portfolio path quality

We include Transformer to show a rigorous negative result. "More complex model" was not a guaranteed upgrade in this regime.

Transformer added complexity and runtime without consistent enough gains — GRU remained the practical winner for this dataset and problem size.

The framework is generalizable in structure, but IPO-specific weaknesses (unstable tails, sector metadata noise) are the binding constraint on performance.
New Experiment · Online Update Policy

Online Update Behavior: Benefits & Side Effects

What We Tested

  • Update (retrain on growing window) vs frozen weights
  • Update frequency: daily, weekly, monthly
  • Threshold-based policy: only update when validation δ exceeds cutoff
  • Turnover cost impact across update schedules

Benefits Observed

  • Online model: Ann. Ret 20.04% vs Market 11.02%
  • Updates improved adaptation to 2020–2022 regime shifts
  • Validation-delta threshold reduced unnecessary retraining

Side Effects

  • Increased turnover (hence cost) near update events
  • Some updates added noise rather than signal
  • Higher sensitivity to data-quality issues at update boundaries
  • Sharpe 1.03 (online) vs 2.53 (val) — longer horizon is harder

Summary & Q&A Backup

What Worked · What Was Mixed · What Regressed

Worked
  • Risk/diversification shaping — prevented concentration collapse
  • Gradient clipping — stable optimization without loss spikes
  • Sector-aware decomposition — deep diagnostic visibility
  • Path-aware evaluation (Sharpe, Sortino, drawdown) over loss-only selection
  • Walk-forward framing — defensible, honest claims
  • Multi-term loss — largest single performance lever
  • Online model beats market baseline: 20.04% vs 11.02% ann. return
Mixed / Limited
  • LSTM: marginal improvement over GRU — conditional on objective calibration
  • Context window tuning — incremental unless paired with objective changes
  • Online updates — useful in trending regimes, noisy/costly otherwise
  • Some multisector settings — better structure, worse data-quality fragility
  • Benchmark-relative terms — helped directionally, hard to calibrate
Failed / Regressed
  • Transformer — added complexity without consistent gains in this regime
  • Objective–selection mismatch — loss improved but path quality did not
  • Materials sector — Sharpe −0.01, MaxDD −42% under IPO concentration
  • Over-regularization — killed useful signal, produced static allocations
  • Some policy network variants underperformed materially

Biggest contribution: not "we found one magic model."

We built a more rigorous optimization workflow — better objective shaping, better checkpoint criteria, better evaluation design, and more honest reporting of failures. That is what makes the project technically stronger.

Q&A: Why loss update? Literature identified tail risk + objective mismatch → explicit loss terms. Why not pick best model? Objective alignment had larger impact than architecture alone. How avoid overfitting? Walk-forward + future-only evaluation + transparent regression reporting.
